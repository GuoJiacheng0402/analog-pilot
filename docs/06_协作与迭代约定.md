# 06 · 协作与迭代约定

本文档关注工作方式。在与 AI 协作推进这类周期长、易出错、易丢失进度的 IC 流程时，以下几条
约定可显著降低出错概率。

---

## 1. Co-pilot 协作模式：人负责方向，AI 负责执行

- **由人负责**：工程判断、设计方向、取舍权衡（如「为某一指标可接受多大的代价」）、验收口径。
- **由 AI 负责**：计算、运行仿真、扫参、整理数据、编写脚本、排查报错、维护日志。
- **关键约定**：要求 AI **通过 SKILL 读取 PDK 的真值**（CDF 参数名、合法取值、器件几何），
  而非凭训练记忆推断。一条有效的提示是：「先 dump 确认，再动手」。

> AI 可能给出一个看似合理但实际错误的参数名或函数签名。要求其先读取真值，可避免大量
> 「修改许久后才发现参数名有误」的反复。

---

## 2. 小步可回退 + 双（三）层备份

IC 流程中每一步都可能破坏上一步的成果（误删 shape、改错参数、覆盖网表）。约定：

- **每到一个里程碑即备份**：DRC 清零、LVS CORRECT、PEX 成功、dummy 接地/接电源、后仿
  通过等，逐一保存。
- **三层备份**：
  1. 本地：脚本与日志打包 `tar`（带时间戳）；
  2. 远端：OA cellview 与 Calibre work dir 打包；
  3. 将远端包同步回本地（`rsync`），异地保存一份。
- **命名带时间戳**：`backup-<阶段>-YYYYMMDD-HHMM.tgz`，便于回滚至任意里程碑。

```bash
# 示例：里程碑备份
ts=$(date +%Y%m%d-%H%M)
rsync -avz <学号>@<服务器>:'~/Design/CSMC05_M3' ~/ic-backup-$ts/Design/
rsync -avz <学号>@<服务器>:'~/work_opa_*'       ~/ic-backup-$ts/work/
tar czf ~/ic-backup-$ts/scripts-$ts.tgz ./tools ./docs
```

---

## 3. 维护「当前状态」块，应对上下文丢失

与 AI 长时间对话时，上下文会被压缩，AI 可能丢失先前的结论。对策：在工作日志（或交给
AI 的文档）**顶部**维护一个固定格式的状态块，每次接手前先阅读、每有新结论即写回：

```markdown
## 当前状态（每次先阅读）
- 硬验证状态：DRC 0(0) 通过 / LVS CORRECT / PEX full R+C xRC Errors=0 / 后仿 5/6
- 当前目标：提升 SR-
- 进行中：扫描 Rz（5K→7K），netlist 位于 work_opa_pex_r-plus-c_*
- 桥接状态：已 start（默认 profile）
- 待办：① 复测 PM ② 备份当前里程碑
- 已确认的注意事项：fw 必须设置；TC 使用启动瞬态；修改 schematic 前先关闭 maestro
```

> 该习惯对人同样有效：间隔数日后重新介入时，先阅读状态块即可快速恢复上下文。

---

## 4. 新会话起始信息

每开启一个新对话，建议将 [AGENTS.md 第 6 节](../AGENTS.md) 的起始信息模板提供给 AI，使其
先阅读 `AGENTS.md` 与 `docs/00_总览与导航.md`，并先执行 `apilot status` 确认连接。
将当前阶段、目标、桥接状态简要交代清楚，可减少大量试探性操作。

---

## 5. 工作产物组织

建议在服务器上建立命名中性的「现场验收」入口，而非将临时脚本散落各处：

```
~/<电路>_verify/
├── run_<电路>_verify.sh        # 入口：每次新建带时间戳的 run 目录
├── <电路>_verify.py            # 重生 testbench → 调用本地 Spectre → 解析本次 PSF
└── runs/run_YYYYMMDD-HHMMSS/   # 每次运行的全部产物（不复用旧结果）
```

本项目 [`tools/postlayout-verify/`](../tools/postlayout-verify/) 即为该结构的通用化实现，
可直接修改 config 后使用。

---

## 6. 截图取证

撰写实验报告或验证服务器状态时，使用
[`tools/grab_vnc_screenshot.sh`](../tools/grab_vnc_screenshot.sh) 经 SSH 将服务器 VNC
桌面抓取至本机（基于 `gnome-screenshot`，这是微电子学院服务器上可靠的截图方式，原因见
[常见问题与排错 H 节](02_常见问题与排错.md#h-截图--工具链)）。

---

## 7. 小结

> **读取真值、小步推进、定期备份、维护状态。** 落实这四点，可显著降低 AI 协作下 IC 流程的
> 出错与返工概率。
