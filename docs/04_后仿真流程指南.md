# 04 · 后仿真流程指南

本文档说明从版图到「前后仿对比表」的完整流程：DRC → LVS → 两步 PEX → Spectre 后仿。
本项目的 [`tools/postlayout-verify/`](../tools/postlayout-verify/) 提供了一套**配置驱动**
的复现脚本，将「从 PEX 网表到指标 CSV」这一段自动化。

> 面向 AI 智能体的对应版本：[`skills/postlayout-verify/SKILL.md`](../skills/postlayout-verify/SKILL.md)。

---

## 0. 流程概览

```
版图  →  DRC clean(0)  →  LVS CORRECT  →  PEX① no-R/C  →  前后仿对齐  →  PEX② full R+C  →  6 项后仿  →  前后仿对比表
                                          (验证流程正确)    (应与前仿一致)   (评估寄生)
```

每一步均依赖上一步的产物。**原则：LVS 未 CORRECT 之前不应进入 PEX。**

---

## 1. 两步 PEX（课程要求，需理解其意义）

| 步骤 | 提取格式 | 目的 | 验收标准 |
|---|---|---|---|
| **第一步** | **NO R/C**（SIMPLE，仅验证连接、不提取寄生） | 验证 **LVS/PEX 流程与 PDK 文件配置** 是否正确 | 前仿与 "NO R/C" 后仿结果应**完全一致**；不一致说明流程或配置存在问题 |
| **第二步** | **完整 R+C** | 评估寄生对性能的实际影响 | 第一步通过后方可进行 |

**关键认知**：第一步 no-R/C **并非评估性能**，而是验证抽取流程的正确性。它与前仿一致，
方能证明后续 full R+C 所见的差异确实来自寄生，而非流程错误。

---

## 2. 工作目录命名约定

为使脚本能稳定定位产物且互不干扰，约定按 `work_<电路>_<阶段>_<cell>` 命名：

```
work_<opa|bgr>_drc_<cell>/          # DRC：<cell>.gds, result.summary
work_<opa|bgr>_lvs_<cell>/          # LVS：<cell>.cdl(.spice), lvs.rep, lvs.rep.sp
work_<opa|bgr>_pex_no-rc_<cell>/    # PEX no-R/C：<cell>.norc.simple.scs
work_<opa|bgr>_pex_r-plus-c_<cell>/ # PEX full R+C：<cell>.pex.netlist (+ .pex/.pxi), gui_batch.log
```

进行 BGR 时一律使用 `work_bgr_*` 前缀，**避免污染**运放的 `work_opa_*` 目录（见
[常见问题与排错 I4](02_常见问题与排错.md#i4-不动已交付的-baseline新功能建独立-cell)）。

## 3. 各阶段执行步骤

### DRC
1. 使用 Cadence `strmout` 将 OA layout 导出为 GDS（**不应使用 SKILL 的 `pipoStreamOut`**，会崩溃）：
   ```
   strmout -library <设计库> -topCell <cell> -view layout -strmFile <cell>.gds \
           -dbuPerUU 1000 -scale 1.0 -runDir <work>
   ```
2. deck 中含 `CELLNAME` 占位符，用 `sed 's|CELLNAME|<cell>|g'` 替换；deck 自带 `LAYOUT PRIMARY`，可直接运行、无需 runset。
3. `cd <work> && calibre -drc -hier -64 <cell>.drc.rules`
4. 解析 `result.summary` 中的 `TOTAL DRC Results Generated: N (M)`，目标为 `0 (0)`。

> DRC 迭代节奏与具体 rule 修复方法见 [常见问题与排错 D 节](02_常见问题与排错.md#d-calibre-drc)。

### LVS
1. 重新 `strmout` 刷新 GDS（携带最新 label/pin）。
2. 用 `sed CELLNAME` 替换 LVS deck。
3. **deck 期望 `.spice` 后缀**，而非 `.cdl` → `cp <cell>.cdl <cell>.spice`。
4. `cd <work> && calibre -lvs -hier -64 <cell>.lvs.rules`
5. 查看 `lvs.rep` 确认 `CORRECT`/`INCORRECT`；调试时查看 `lvs.rep.sp`（抽取网表）。

> source CDL 必须使用 deck subtype 名（NP/NN/RL/H1/CP/P1），LVS 调试方法见
> [常见问题与排错 E 节](02_常见问题与排错.md#e-calibre-lvs)。

### PEX（两步，使用 GUI runset batch）
**不应手写 SVRF deck**（见 [常见问题与排错 F1](02_常见问题与排错.md#f1-手写-pex-netlist-各种报错inppssspc-系列甚至-sigsegv)）。流程：
1. 将 GDS + CDL(`.spice`) 软链至 work dir。
2. 复制课程 runset 模板，用脚本逐行修改字段（指向本设计的 GDS/CDL/deck/work dir，
   `pexPexNetlistType=RCC`、`pexPexNetlistFormat=SPECTRE`，ground 设为 `VSS`）。
3. `calibre -gui -pex -runset pex_gui.runset -batch`（内含 LVS → xRC PDB → FMT）。
4. **no-R/C 模式**额外步骤：删除 GUI 生成 deck 中的 `PEX NETLIST` 行，插入
   `PEX NETLIST SIMPLE "<out>" SPECTRE 1 SOURCENAMES SEPARATOR "/"`，再
   `calibre -xrc -fmt -simple -64 ...`。

> ground 警告的规范修复方法见 [常见问题与排错 F2](02_常见问题与排错.md#f2-pex-警告-no-ground-net-name-defined--0-will-be-used)。

### Spectre 后仿
后仿**直接使用 Spectre netlist 驱动**，不经 ADE GUI（以规避 maestro 阻塞）。要点：
- PEX 网表中 `rpoly2/rhr1k/cpip` 需映射为内建 `resistor/capacitor`（在仿真副本上操作，不修改原网表）。
- PEX 子电路端口顺序与 schematic 不同，testbench 按各自顺序实例化。
- full R+C 主网表的相对 include 需解析为绝对路径。
- 上述处理均已封装在 [`tools/postlayout-verify/`](../tools/postlayout-verify/) 中。

## 4. 使用 postlayout-verify 工具复现

`tools/postlayout-verify/` 提供两个配置驱动的 driver（纯标准库，运行于服务器）：

```bash
cd tools/postlayout-verify
cp configs/opa.example.json configs/opa.local.json    # 修改为实际的 cell 名与路径
cp configs/bgr.example.json configs/bgr.local.json

# 运放：运行 full R+C 一档
./run_opa_verify.sh --config configs/opa.local.json --variants pex_rc
# 运放：一次运行全部三档（schematic / no-R/C / full R+C）
./run_opa_verify.sh --config configs/opa.local.json

# BGR：full R+C 启动瞬态温扫
./run_bgr_verify.sh --config configs/bgr.local.json --variants pex_rc
# BGR：在流程内先重跑 Calibre PEX，再进行温扫
./run_bgr_verify.sh --config configs/bgr.local.json --variants pex_rc --rerun-pex
```

每次运行都**新建一个带时间戳的 run 目录**，重新生成 testbench、重新调用 Spectre、重新
解析本次 PSF——**不复用任何旧 CSV/日志/截图/raw**。产物：
- `opa_live_metrics.csv` —— 运放六项指标（Adc/GBW/PM/Idc/SR+/SR-），每个 variant 一行。
- `bgr_live_metrics.csv` —— BGR 温度系数汇总。
- `temperature_points.csv` —— BGR 逐温度点的 Vref/Idd 等。
- `all_waveform_points.csv` —— 用于绘图与审计的长表。
- 以及产生上述数据的 `.scs` testbench、Spectre 日志、PSF raw 目录。

> 配置项（model 路径、spectre 路径、各 variant 的 netlist、端口顺序、license 等）详见
> [`tools/postlayout-verify/configs/*.example.json`](../tools/postlayout-verify/configs/) 与
> [postlayout-verify skill](../skills/postlayout-verify/SKILL.md)。
> `cds_lic_file` / `mgls_license_file` 支持 `{hostname}` 占位符，运行时自动替换为当前主机名。

## 5. 验收口径与报告建议

- **运放**：以 **full R+C 后仿 6/6 通过**为准（Adc / GBW / PM / Idc / SR+ / SR-）。
- **BGR**：以 **full R+C 启动瞬态温扫的 TC** 为准（如 `< 10 ppm/°C`），定义
  `TC = ΔVref / (mean(Vref) × ΔT) × 1e6`。
- 报告中给出**前后仿对比表**，并以文字说明 layout-driven 调参（post-layout iteration）的
  物理原因（如寄生分压对工作点电流的反向修正）。相比强行使前后仿电路完全一致，此做法更
  符合工程实际。
