## A. 连接 / 环境 / 部署

### A1. `import apilot` 报 ModuleNotFoundError（conda 干扰）
- **现象**：venv 已激活、`pip install -e .` 也成功，但仍报 `ModuleNotFoundError: No module named 'apilot'`。
- **原因**：终端默认 `conda activate base`，`python` 虽指向 venv，但 `sys.path` 仍使用 anaconda 的 stdlib，editable 安装写入的 `.pth` 未被 anaconda 的 site.py 处理。
- **解决**：临时方案 `export PYTHONPATH=<repo>/src`；根治方案为注释掉 `~/.zshrc` 中的 `conda activate base`，或定义别名 `conda deactivate && source .venv/bin/activate`。

### A2. `apilot start` 报 "No Python interpreter found ... Detection output: ''"
- **现象**：远端存在 python3，但 start 探测不到，输出为空字符串。
- **原因**：启用 SSH **ControlMaster** 复用连接时，探测命令经 `sh -l -s` 的 stdin 传递存在时序问题，返回为空；手动执行 `ssh ... 'sh -l -s'` 传入相同命令则正常。
- **解决**：在 `.env` 中设置 **`APILOT_DISABLE_CONTROL_MASTER=true`**。代价仅为每次新会话首次握手增加 1~2 秒。**该项为本环境的必设项**，模板已默认开启。

### A3. 启动 Virtuoso 报 `error while loading shared libraries: libap_sh.so`
- **原因**：未 source Cadence 环境即直接运行 `virtuoso`。
- **解决**：先 `source /SM01/eda/env_set/bashrc_cds`，再 `source /SM01/eda/env_set/bashrc_mentor`，**顺序固定**（先 cds 后 mentor），缺一不可。注意是这两个学校提供的环境脚本，而非 `~/.bashrc_cds`。

### A4. source 环境后 git / curl / openssl 行为异常
- **原因**：`bashrc_cds` 强制设置了 `LD_PRELOAD=/usr/lib64/libssl.so.1.0.2k:/usr/lib64/libcrypto.so.1.0.2k`（IC618 依赖旧版 OpenSSL），影响整个 shell。
- **解决**：用于启动 Virtuoso 的 shell 在 source 后应专用于启动 Virtuoso，其他操作另开终端；**不应**将 `source bashrc_cds` 写入 `~/.bashrc`（会影响所有会话，并干扰桥接的免密与探测）。

### A5. 重启后已部署的 daemon 文件丢失
- **原因**：scratch 默认位于 `/tmp`，服务器重启会清空 `/tmp`。
- **解决**：在 `.env` 中设置 `APILOT_REMOTE_SCRATCH_ROOT=/SM01/home/<grade>/<学号>/.apilot`（家目录固定路径），模板已默认。

### A6. `APILOT_CADENCE_CSHRC` 填写 `bashrc_cds` → 桥接报 csh 解析错误
- **原因**：桥接以 `csh -c 'source $APILOT_CADENCE_CSHRC; env'` 加载该文件，要求 **csh 语法**；学校的 `bashrc_cds` 为 **bash 语法**（`export FOO=bar`），csh 解析报错。
- **解决**：
  - 纯 SKILL 模式（最常用）：**`APILOT_CADENCE_CSHRC` 留空**。SKILL 在 Virtuoso 主进程内执行，不经远端登录 shell，无需此项。
  - 使用 Spectre 独立通道时：基于 [`templates/cadence_env.csh.example`](templates/cadence_env.csh.example) 改写为 csh 版（`export` → `setenv`），并将 `APILOT_CADENCE_CSHRC` 指向它。

### A7. macOS 自带 rsync 版本过旧
- **现象**：`rsync --info=progress2 ...` 或多源参数报错；系统自带 rsync 为 2.6.9（2006 年）。
- **解决**：`brew install rsync`（使用 3.x 版本）。

### A8. `~/.login_log` 中出现本地机器的 IP
- **现象**：若服务器 `~/.bashrc` 中存在登录监控脚本，桥接建立 SSH 隧道时也会触发它，从而记录本地机器 IP。
- **结论**：此为**预期行为**，仅为审计记录，不影响功能，无需处理。

## B. SKILL / GUI 对话框阻塞

> **共性原因**：桥接的 SKILL 通道是 Virtuoso 主进程内的单线程 IPC handler。任何使 Virtuoso
> 弹出**模态对话框/表单**的 SKILL 都会使 handler 停留等待用户操作，导致**后续所有
> `execute_skill` 超时**，`status` 显示 `[daemon] NO RESPONSE`。服务器通常未安装
> `xdotool/wmctrl/xte`，本地也常未配置 X11 转发，因此 `dismiss-dialog` 多数情况下不可用。
> **首要处理动作：在 VNC 桌面检查是否有未关闭的对话框，并手动关闭。**

### B1. 使用 `schViewToView` 八参形式生成 symbol → 弹出 `auCreateSymbol` 表单导致阻塞
- **解决**：使用**无表单**的批量 API：
  ```python
  client.execute_skill(
      f'schPinListToSymbol("{lib}" "{cell}" "symbol" '
      f'schSchemToPinList("{lib}" "{cell}" "schematic"))')
  ```
  一次即可成功，不弹出任何表单。**本 PDK 不应手动构建 symbol，应直接使用 `schPinListToSymbol`。**

### B2. 修改正在被 maestro 引用的 schematic → 弹出 "re-netlist?" 导致阻塞
- **原则**：**maestro session 处于活动状态时不应修改 schematic。** 正确顺序：建立 schematic
  → 设置全部参数 → 再 `maeOpenSetup`。（这是 daemon 阻塞最常见的原因之一。）

### B3. `maeRunSimulation` 触发 ASSEMBLER-9039 "Update and Run" 对话框导致阻塞
- **无效尝试**：清理 / 删除 maestro 目录、通过 SKILL 关闭 `checkFromSetupDB`——仍触发 9039。
- **可行的规避方法**：fire-and-forget + marker 文件。在 SKILL 回调中执行 `system("echo done > /tmp/marker_xxx")`，`execute_skill(..., timeout=5)` 必然超时但 SKILL 已进入队列 → 由用户在 VNC 中点击 "Update and Run"（而非 Cancel）→ 仿真完成后触发 marker → Python 轮询 `cat /tmp/marker`。
- **更稳妥的根治方法**：先将 schematic 真正存盘同步（`dbSave(cv)` + `hiSaveAllCellViews`），setupDB-changed 检查即被跳过，9039 不再弹出。
- **推荐做法**：与其与 maestro GUI 反复对抗，**后仿应直接使用 Spectre netlist**（见 G 节与 [05 方法学](05_仿真与验证方法学.md)）。

### B4. 同一 cell 打开了十余个 schematic editor → 切换 cell 超时 + ASSEMBLER-8130
- **教训**：不应将同一 cell 反复打开多次。X11 发送 Esc/Return/Alt+F4/坐标点击均无法关闭此类对话框；强制关闭还可能一并关闭 Virtuoso 主窗口，导致进程退出，需重启并重新 load。

### B5. `dismiss-dialog` 不可用
- **原因**：该功能需要本地 X11 转发至服务器，且服务器安装 xdotool，本环境通常两者均不具备。
- **解决**：由用户在 VNC 中手动关闭对话框。**面向 AI 的提示**：一旦发现 `NO RESPONSE`，应首先询问用户「VNC 中是否有未关闭的 Cadence 表单」。

### B6. `t` 为受保护变量
- **现象**：在 `setof(t list pred)` 等表达式中将循环变量命名为 `t` → `setq/set: Variable is protected`，且报错显示为 `line N at EOF`（**表现为括号不匹配**，具有误导性）。
- **解决**：循环变量不应命名为 `t`，应改用 `xt` 等。

### B7. 其他 SKILL parser 注意事项
- `procedure(...)` 不能嵌套于 `let((...) ...)` 中 → 应改用 `foreach(spec list let((vars) body))`。
- 赋值号两侧需有空格：应写 `cc = ddGetObj(...)`，写成 `cc=ddGetObj(...)` 会报 column 29 syntax error。
- 本 PDK 中 schematic 的 viewType 为 `"schematic"`，而非 `"schematicSchematic"`。
- 如需保留 SKILL 文件以便调试（桥接以 `load()` 执行后会自动删除）：使用 `runner.upload_text(SKILL, "/tmp/foo.il")` + `execute_skill('load("/tmp/foo.il")')`。

## C. CSMC PDK / 仿真语义

> 本类问题为 CSMC05_M3 工艺特有的语义陷阱。**完整、带示例的版本见
> [03 · CSMC05_M3 PDK 备忘](03_CSMC05_PDK备忘.md) 与 [skills/csmc-pdk](../skills/csmc-pdk/SKILL.md)。**
> 此处仅列出几条「会导致 DC 工作点错误且难以定位」的提示。

### C1.（重要）MOS 仅设置 `w` 而未设置 `fw` → 仿真中均为最小尺寸
- CSMC 的 `mn/mp` 中 `w` 仅作显示、**不进入 netlist**；进入 netlist 的是 `fw`（finger width），`simW = fw × fingers`。仅设置 `w` 会导致所有 MOS 为默认最小尺寸，DC 工作点全部错误。
- **解决**：通过 `client.execute(...)` 下发设置 `w/l/fw/fingers` 的 SKILL，**必须包含 `fw`**（仅设 `w/l` 会落到最小尺寸）。

### C2. 电阻仅传 `r="5K"` → 几何不变、抽取阻值错误
- `rpoly2/rhr1k` 传入字符串 `r=` 仅写入属性、不改变几何，Calibre 按默认几何抽取。
- **解决**：使用 `calculatedParam="Length"` + `segL/segW`（浮点 SI 米）+ `segments/ssegs`。

### C3. vsource pulse 高低电平写成 `v1/v2` → 被静默忽略、波形不变化
- 字段为 `val0`/`val1`，而非 hspice 风格的 `v1/v2`（误用不报错，但 pulse 不切换）。
- pulse 参数名为 `td/tr/tf/twidth/per`，而非 delay/rise/fall/width/period。

### C4. BJT 双 section include 冲突
- 模型需 include `section=tt`（MOS）与 `section=biptypical`（BJT）；**不应**再 include `section=bip` → 会报 `Model qvp5 already defined`（`biptypical` 已嵌套包含带正确参数的 `bip`）。

### C5. 自行计算相位得到数千度的错误值
- `(180/π)*phase(...)` 会得到 -6300° 等错误值，应使用 `phaseDegUnwrapped(VF("..."))`。

> 更多内容（缺模型 `SFE-23`、表达式转义、AC 源 srcType 等）见 [03 PDK 备忘](03_CSMC05_PDK备忘.md)。

## D. Calibre DRC

> 修复节奏：定位 `result.summary` 中出现频次最高的 rule，**按金属层自下而上修复**（先
> W1/contact，再 A1/W2，再 A2/W3/T3）。需区分硬规则与 `soft_check_*`（guard ring 的软警告
> 通常是合理的）。

### D1. 手绘 gate contact 反复被判违例（W1.g / GT.* 等）
- **原因**：手绘 gate access contact 会被 deck 派生为违例，仅调整偏移只能减少而无法清零。
- **解决**：改用 **PDK 原生 gate contact**——`mn/mp` PCell 传入 `connectGates="Top"`，由 PDK 生成合法的 `G`/`A1` pin，走线仅从该 pin 上 via。

### D2. via 尺寸必须严格匹配 PDK（W2/W3 系列）
- **解决**：**W2 via 严格为 `0.55×0.55`，W3 via 为 `0.6×0.6`**；guard ring 上添加 via 应采用**稀疏**方案——每簇 ≤2 个 via stack、间距 ≥4µm、避开既有走线。密集 via 一次可引入数十条违例。

### D3. `offgrid_check` / `IM.h` 大量出现
- **原因**：物理参数使用了非网格值（如 `22.19986u`、`segL=6.633u`）。
- **解决**：所有物理参数**量化到 0.01µm 网格**（`22.20u`、`6.63u`、`1.49u` 等）。
- 此外：`rhr1k` 段长过小会触发 `IM.a`（IM 最小 2.0µm），应将 `L` 提高至 ≥2.0µm 并调整 `W` 以保持目标阻值。

### D4. `connectSD="Both"` 对多指管的 S/D strap 造成意外短路（DRC 不报、LVS 报）
- **解决**：将易短路的多指管由 `fingers=2, fw=3.5u` 改为 `fingers=1, fw=7u`（总宽不变、一阶电气等价），消除该低位 source strap。

## E. Calibre LVS

> 思路：① source CDL 使用 **deck 的 subtype 名**；② 查看 `lvs.rep.sp`（抽取网表）与
> `lvs.rep` 的 detailed connections 定位真实网络；③ 通过「隔离实验」（临时省略某条 strap）
> 二分定位短路源。**LVS 未 CORRECT 之前不应进入 PEX。**

### E1. source CDL 使用仿真模型名 → 器件类型不匹配
- **现象**：CDL 使用 `mn/mp/rpoly2/rhr1k/cpip`，而 layout 抽取为 `NP/NN`、`RL/H1`、`CP`，无法合并。
- **解决**：source CDL 一律使用 **deck subtype 字母**：MOS `NP/NN`；电阻 `RL`（rpoly2）/`H1`（rhr1k）；电容 `CP`（cpip）；PNP `P1`（`area=25p`，对应 deck 中 `DEVICE Q(P1) _qvp5 ...`、`NETLIST MODEL "qvp5"`）。

### E2. DRC 通过，LVS 却整片短路（同层长 bus 横穿）
- **现象**：一条 A2 或 T3 的长 bus 横穿一排管子的同层 patch 或一串竖线 → 电气短路；DRC 不报。
- **解决**：将长 bus 改走另一层（如 A2 → T3 trunk），局部用 via 接入，避免同层交叉。**注意：DRC 通过不等于无电气短路。**

### E3. 高阻输入网与参考/电源网同层交叉 → 被抽取为同一网
- **解决**：**电源/地/参考网（VDD/VSS/VREF）保留在 T3，高阻输入网（运放 +/- 输入）走 A2**，以 `A1+W2+A2` 接入；必要时用一小段短 T3 bridge 跨越另一条输入网后再回到 A2。

### E4. guard ring A1 环浮空 → body 被抽取为未命名网
- **解决**：为每个 island 的 guard ring 添加一个远离信号的稀疏 `W2+A2+W3` via 栈，将 A1 环接至 VDD/VSS rail（PMOS body → VDD，NMOS body → VSS）。

### E5. dummy 管的 G/D/S 浮空（违反接地/接电源要求，且 PEX 出现浮节点）
- **解决**：为每个 dummy 用一块**局部 A1 plate** 覆盖其 D/G/S access 并直接接触已接电源的 guard-ring A1（PMOS dummy → VDD，NMOS dummy → VSS）；source CDL 中将 dummy 四端均写至电源/地。

### E6. 电容/电阻抽取值与 source 存在微小差异 → property error
- **解决**：仅修改 **source CDL** 为实测抽取值（如电容 `c=354.845f`），**不修改版图**。

### E7. BJT 面积比用 `m` 实现 → LVS 实例数/面积不匹配
- **原因**：`qvp5` 的 `m` 仅服务于仿真，**layout 不放大几何**。
- **解决**：面积比 `n` 通过**并联 n 个单位 `qvp5`** 实现，source CDL 同样写入 n 个并联实例。BJT 的 emitter 从中心 A1 直接 via 至上层金属（避免被环形 base/collector 短路），T3 跨越 ring 时不下 via。

## F. Calibre PEX

> 结论先行：**不应手写 SVRF PEX deck**（问题繁多，甚至会导致 calibre SIGSEGV）。应使用
> **Calibre Interactive 的 GUI runset + `-batch`**，由工具自动生成合法语句。

### F1. 手写 `PEX NETLIST ...` 各种报错（INP/PSS/SPC 系列，甚至 SIGSEGV）
- 典型情况：`PEX EXTRACT MAPPED` → INP7；`PEX REPORT` → PSS2；自带 `LAYOUT PRIMARY` 冲突 → SPC1；`PEX NETLIST CAPACITOR/RESISTOR NONE` → **SIGSEGV(rc=139)**；`PEX NETLIST "out" SPICE 1 LAYOUT` → **SIGSEGV**；缺 XRC option → `MASK SVDB DIRECTORY statement does not contain the XRC option` 等。
- **GUI 自动生成的可用语句**如下（仅供理解，不应手动编写）：
  ```
  PEX NETLIST "<out>" SPECTRE 1 SOURCENAMES SEPARATOR "/"
  MASK SVDB DIRECTORY "<svdb>" QUERY XRC
  ```
- **解决**：使用 `calibre -gui -pex -runset pex_gui.runset -batch`（可由本项目的
  [postlayout-verify](../skills/postlayout-verify/SKILL.md) 脚本自动修改 runset 字段）。

### F2. PEX 警告 `No ground net name defined ... "0" will be used`
- **原因**：课程默认 `pex.runset` 中 `*pexPexGroundName: 0`、`*pexPexGroundNameValue:` 为空。
- **解决**：在 GUI runset 中设置 `pexPexGroundName=1`、`pexPexGroundNameValue=VSS`（BGR 还可添加 `pexGroundNames=VSS`、`pexPEXGroundNetNames=VSS`），生成 `... SOURCENAMES GROUND "VSS" SEPARATOR "/"`，警告消失。
- **边界**：no-R/C 的 **SIMPLE formatter 不支持 `GROUND` 关键字**，该 `PEX NETLIST SIMPLE ...` 语句应保持原样、不添加 `GROUND`（它复用 GUI 生成的 svdb/PDB，最终亦无警告）。

## G. Spectre 后仿

### G1. PEX 网表中 `rpoly2/rhr1k/cpip` 在 Spectre 中未定义
- **解决**：生成**仿真专用副本**，将其映射为内建 `resistor/capacitor`（如 `) rpoly2 r=` → `) resistor r=`），**不修改原始 PEX 网表**。需处理反斜杠续行（`\` 后接空行会破坏参数行）。

### G2. PEX 子电路端口顺序与 schematic 不同
- **现象**：PEX 为 `subckt opa (VDD VSS VOUT VIN1 VIN2)`，schematic 为 `(VDD VIN1 VIN2 VOUT VSS)`；BGR PEX 为 `(VREF VSS VDD)`。
- **解决**：testbench 按各自端口顺序实例化（不同 variant 使用不同顺序）。

### G3. full R+C 主网表的相对 include 找不到 companion 文件
- **报错**：`Cannot open the input file '...pex.netlist.pex'` / `'....pxi'`。
- **原因**：full R+C 主网表 include 两个 companion 文件（`.pex` 与 `.pxi`），为**相对路径**；当主网表被复制到其他 run 目录、在该目录运行 Spectre 时即无法找到。
- **解决**：将相对 include 改写为源 PEX workdir 下的**绝对路径**（本项目 `bgr_verify.py` 的 `sanitize_pex_for_spectre` 已实现）。

### G4. 缺模型 `SFE-23: undefined model 'mp'/'mn'`
- **解决**：在 `set_env_option` 或 netlist 中 include `s05mixdtssa01v12.scs` 的 `section=tt`（详见 [03 PDK 备忘](03_CSMC05_PDK备忘.md)）。

### G5. 误以为在运行 Spectre，实际在运行 HSPICE
- **现象**：ADE 标题包含 "spectre" 字样，但 run dir 为 `.../hspiceD/schematic`，`si.env` 中 `simSimulator="hspiceD"`；`psf/hspice.out` 报 `hspice: Command not found.`，切换 HSPICE 环境后又报 `.lib entry "tt" cannot be found`、`.AC DEC 10` 缺 start/stop。
- **结论**：**前仿与报告截图应优先使用 Spectre**。在 ADE L 中先 `Setup → Simulator/Directory/Host` 选择 `spectre`，确认 run dir 为 `.../spectre/schematic`，模型使用 `.scs` 的 `section=tt`。仅在确需运行 HSPICE 时才切换 Synopsys 环境并使用 `.lib`。

### G6. BGR 纯 DC 温扫落入低压零电流分支（无效结果）
- **现象**：no-R/C 与 full R+C 的 DC 温扫均收敛至 `VREF≈0.05~0.44V`，TC 表现为数千 ppm/°C。
- **原因**：BGR 无显式启动电路时，纯 DC sweep 易收敛至低电流分支。
- **解决**：**使用启动瞬态**——每个温度点将 VDD 从 0V ramp 至额定（如 2µs 上升），瞬态运行至稳态（如 80µs），取末段（如末 10µs）平均，**逐点独立上电、不读取旧 OP**。详见 [05 方法学](05_仿真与验证方法学.md)。

## I. 方法论与流程注意事项

### I1. 修改 PCell 参数后未重新 dump 几何 → 走线使用旧坐标，"实验结论"失真
- **教训**：标准 pipeline **必须包含「重新 dump PCell 真实几何」这一步**，否则走线按旧落点生成，看似已修改实际未生效。

### I2. 试图使「前仿电路 == 后仿电路」反而无法满足 spec
- **现象**：将 schematic 改为与 finalfix 版相同的尺寸后，前仿某指标超标。
- **物理解释（可写入报告）**：后仿因互连寄生电阻在 VDD/VSS 干线上分压，工作点电流反而更低、更易达标——这是寄生对工作点的反向修正。
- **做法**：采用工业界常规的 **post-layout iteration**——保留前仿 baseline（schematic）与后仿修正版（layout），并以文字说明这是 layout-driven 调参，而非强行使两者电路完全一致。

### I3. 以 no-R/C 与 full R+C 的对照进行寄生归因
- 两者结果**高度一致** → 问题并非由寄生引起（可能是收敛分支或连接问题）；两者差异较大 → 方为寄生影响。该对照可快速排除「误判为寄生」的情况。

### I4. 不动已交付的 baseline，新功能建独立 cell
- 进行 BGR 设计时不修改已验收的运放 cell，而是**新建独立 cell** 将运放作为宏实例化；Calibre 工作目录也一律使用新前缀（如 `work_bgr_*`），避免污染运放的目录。
